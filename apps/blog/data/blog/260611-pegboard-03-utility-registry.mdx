[1편](/blog/260609-pegboard-01-start)에서 *"페이지 100개의 품질은 페이지가 시스템 안에 들어가는 방식에서 결정된다"* 라고 적었다. 이번 편은 그 시스템의 첫 부품 — **유틸리티 레지스트리** 얘기다.

## 페이지가 늘어나면 무엇이 망가지는가

페이지 18개쯤 됐을 때 처음 이상한 징후가 보였다. 새 페이지를 띄울 때마다 다음 다섯 곳을 동시에 손대야 했다.

```
새 유틸 추가 시 손대야 하는 곳:
├── app/[locale]/<slug>/page.tsx
├── 사이드바 / 네비게이션 (어떤 페이지가 이 도구를 알아야 하나)
├── app/sitemap.ts                 (검색 엔진에 신고)
├── 푸터의 카테고리 목록
├── 홈의 도구 디렉터리 그리드
└── messages/<locale>.json × 7      (i18n)
```

이게 18개에서도 짜증나는데 100개가 되면 *누락이 무조건 생긴다.* 단순한 수학이다. 그리고 누락이 생기는 곳이 묘하게 무서운 곳이다 — 사이트맵에서 빠지면 SEO가 망가지고, 푸터에서 빠지면 페이지가 고아가 되고, e2e 게이트에서 빠지면 [2편의 quicktype 사건](/blog/260610-pegboard-02-static-export-trap)이 또 일어난다.

이 문제의 답은 단순하다: **모든 표면이 한 곳을 읽게 한다.**

## 단일 SoT — `src/utilities/registry.ts`

이 파일이 시스템의 심장이다 (`src/utilities/registry.ts`).

```typescript
export interface UtilityEntry {
  /** URL slug under `app/[locale]/`. Used as the route segment. */
  readonly slug: string;
  /** Category the utility belongs to. Drives nav grouping / category hubs. */
  readonly category: UtilityCategory;
  /** Top-level namespace in each `messages/<locale>.json`. */
  readonly i18nNamespace: string;
  /**
   * Camel-cased prefix used in the `Home` namespace.
   * Resolves to `${prefix}Title` and `${prefix}Description`.
   */
  readonly homeKeyPrefix: string;
  /** ISO date the utility first shipped. Used for ordering / freshness signals. */
  readonly introducedAt: string;
}

export const utilities = [
  {
    slug: 'dummy-text',
    category: 'text',
    i18nNamespace: 'DummyText',
    homeKeyPrefix: 'dummyText',
    introducedAt: '2025-05-10',
  },
  {
    slug: 'base64',
    category: 'convert',
    i18nNamespace: 'Base64',
    homeKeyPrefix: 'base64',
    introducedAt: '2026-05-24',
  },
  // ... 21개의 유틸이 줄지어 등록
] as const satisfies readonly UtilityEntry[];
```

엔트리 하나가 한 유틸의 *불변 정체성*이다. URL 슬러그, 카테고리, i18n 네임스페이스, Home 키 프리픽스, 도입일. 페이지 컴포넌트 자체는 별개로 살지만, 시스템 안에서의 좌표는 이 한 줄로 정해진다.

## 한 줄 추가 → 모든 표면이 따라오는 메커니즘

이 파일 하나가 어떻게 다른 모든 표면을 끌고 다니는지 보자.

```
src/utilities/registry.ts  ← 새 엔트리 한 줄 추가
        │
        ├──→ src/utilities/nav.ts         (네비/푸터 카테고리 그룹)
        │       └──→ PegboardSidebar.tsx  (좌측 전역 사이드바)
        │       └──→ 홈 디렉터리 그리드
        │       └──→ SiteFooter.tsx
        │
        ├──→ src/lib/seo/site.ts          (sitemap, OG, JSON-LD)
        │       └──→ app/sitemap.ts        (모든 로케일 × 모든 유틸)
        │       └──→ app/robots.ts
        │       └──→ <head> hreflang
        │
        ├──→ tests/e2e/utility-smoke.spec.ts  (브라우저 스모크 게이트)
        │       └──→ for (const { slug } of utilities) { test(...) }
        │
        └──→ Pegboard Curator 서브에이전트 (시스템 지도)
                └──→ 새 유틸 배치 추천, 드리프트 스캔
```

엔트리 한 줄을 추가하면 사이드바·푸터·sitemap·robots·hreflang·e2e 스모크가 *그 다음 빌드부터* 새 유틸을 인지한다. 사람이 잊을 일이 없다.

### nav.ts — 카테고리 그룹화

`src/utilities/nav.ts`는 registry를 카테고리 그룹으로 묶어준다.

```typescript
export const utilityNavSpecs = utilities.map((entry) => ({
  slug: entry.slug,
  category: entry.category,
  href: `/${entry.slug}`,
  titleKey: `${entry.homeKeyPrefix}Title` as UtilityHomeTitleKey,
  descriptionKey: `${entry.homeKeyPrefix}Description` as UtilityHomeDescriptionKey,
}));

export function getCategoryGroups(): readonly UtilityCategoryGroup[] {
  const byCategory = new Map<UtilityCategory, UtilityNavSpec[]>();
  for (const spec of utilityNavSpecs) {
    const bucket = byCategory.get(spec.category) ?? [];
    bucket.push(spec);
    byCategory.set(spec.category, bucket);
  }
  return getOrderedCategories()
    .map((category) => ({
      category,
      utilities: byCategory.get(category.id) ?? [],
    }))
    .filter((group) => group.utilities.length > 0);
}
```

마지막 줄 `filter((group) => group.utilities.length > 0)`이 작은 디테일이다. **유틸이 0개인 카테고리는 nav에 안 뜬다.** 즉 카테고리는 *첫 유틸이 등장할 때* 같이 생긴다. 도메인 언어를 정의하는 `CONTEXT.md` 에 한 줄로 박혀 있다 — *"Add a category only when its first Utility ships."*

## 타입 시스템이 거드는 부분

레지스트리는 단순한 데이터가 아니라 **타입의 원천**이기도 하다.

```typescript
export const utilities = [
  // ...
] as const satisfies readonly UtilityEntry[];

export type UtilitySlug = (typeof utilities)[number]['slug'];
export type UtilityHomeKeyPrefix = (typeof utilities)[number]['homeKeyPrefix'];
```

`as const satisfies` 패턴 덕에 `UtilitySlug`는 `'dummy-text' | 'character-count' | 'base64' | ...` 라는 *리터럴 유니온*이 된다. 오타로 `bs64` 라고 적으면 컴파일러가 잡아준다.

이게 nav.ts에서도 빛난다.

```typescript
export type UtilityHomeTitleKey = `${UtilityHomeKeyPrefix}Title`;
export type UtilityHomeDescriptionKey = `${UtilityHomeKeyPrefix}Description`;
```

템플릿 리터럴 타입으로 *모든 유틸의 i18n 키 타입*이 자동으로 derive 된다. `'base64Title' | 'jsonFormatterTitle' | ...` 같은 유니온. `messages/<locale>.json` 에서 이 키가 빠지면 next-intl이 런타임에 missing key를 던지지만, 타입 시스템이 더 빠르게 잡는다.

## ADR-0001 — "왜 registry 인가" 를 박아두는 이유

이 결정 자체를 `docs/adr/0001-utility-registry.md` 의 ADR로 박았다. 짧지만 명확한 이유 두 가지.

> 페이지가 늘어나면 *건당 품질*보다 **시스템 코히어런스의 드리프트**가 더 큰 위험이 된다 — 정보 구조 표류, 카테고리 불일치, 메타데이터 패턴 분기, 고아 페이지.

> nav 등록, sitemap.ts, 카테고리 허브, 내부 링크, Curator 서브에이전트의 시스템 지도 — 모두 같은 메타데이터(슬러그, 카테고리, i18n 네임스페이스)를 필요로 한다.

ADR로 남기는 기준은 9·10편에서 자세히 다루는데, 이 결정은 셋 다 충족한다:

| ADR 기준                | 이 결정의 답                                            |
| ----------------------- | ------------------------------------------------------- |
| Hard to reverse?        | ✅ 한번 100페이지가 다 이 패턴이면 다시 흩는 비용 큼    |
| Surprising w/o context? | ✅ "왜 페이지마다 인라인 배열이 없지?" 미래의 내가 묻는다 |
| Real trade-off?         | ✅ 처음엔 페이지가 단순히 더 무거워 보임 (간접 1단계)   |

## "필수 트리거" — Curator 서브에이전트와 함께 운영

레지스트리만 두면 사람이 손으로 엔트리를 추가할 때 *카테고리 잘못 고르기*, *슬러그 컨벤션 깨기* 같은 휴먼 에러가 또 들어온다. 그래서 AGENTS.md에 강제 트리거를 박았다.

> **Curator trigger (mandatory):** `src/utilities/registry.ts`를 편집하거나, 유틸을 추가하거나, IA/nav/sitemap/카테고리 배치에 영향을 주는 변경을 하기 전에 Pegboard Curator 서브에이전트를 호출하고 그 리포트를 결정 자료로 사용한다.

Curator는 read-only 서브에이전트인데, 레지스트리·glossary·ADR을 자기 컨텍스트로 들고 다음 세 질문에만 답한다:

1. **Placement** — "새 `markdown-to-html` 유틸 어디에 배치? 슬러그·카테고리·형제 유틸 추천."
2. **Change audit** — "이 diff가 사이트 전체 패턴을 깨는가?"
3. **Drift scan** — "레지스트리에 없는데 페이지가 있거나, 반대거나, 일부 로케일에만 있는 i18n 키 찾기."

자세한 건 9편(에이전트 설정)에서 다루지만, **레지스트리가 데이터의 SoT라면 Curator는 그걸 읽고 추론하는 머리**다. 둘이 짝이다.

## 함정 — 레지스트리가 *명목상 SoT*가 되는 순간

ADR-0001에 솔직하게 적어둔 부분이 하나 있다.

> 초기에는 페이지 컴포넌트가 여전히 인라인 배열을 쓰므로 SoT가 *명목상*에 그칠 수 있다. 페이지 마이그레이션 작업이 후속으로 필요하다.

처음에 registry.ts만 만들어두고 페이지마다 `const tools = [...]` 하드코딩이 남아 있으면, 그건 SoT가 아니라 *또 하나의 위치*가 된 거다. 이걸 막으려면 페이지 컴포넌트도 registry를 *진짜로* 소비해야 한다. 마이그레이션이 끝난 시점부터 SoT가 실재한다.

지금은 nav·sitemap·footer·홈 디렉터리·e2e 스모크 전부 registry에서 derive 된다.

## 정리

- 페이지가 늘면 *건당 품질*보다 **시스템 드리프트**가 더 큰 위험이다.
- `src/utilities/registry.ts`가 단일 SoT — nav, sitemap, footer, 카테고리 허브, e2e 게이트, Curator 서브에이전트가 전부 이 파일을 읽는다.
- 엔트리 한 줄 추가 = 사이드바·sitemap·hreflang·브라우저 스모크가 *그 다음 빌드부터* 새 유틸을 안다.
- `as const satisfies` 패턴 덕에 슬러그·홈 키 프리픽스가 *리터럴 유니온 타입*이 되어 오타가 컴파일러에 잡힌다.
- 데이터(레지스트리)와 머리(Curator 서브에이전트)는 짝. 데이터 없이 Curator는 환각, Curator 없이 레지스트리는 손 컨벤션.
- 진짜 SoT가 되려면 *모든 소비자가 derive*해야 한다. 페이지 컴포넌트에 인라인 배열이 남아 있으면 SoT는 명목상에 그친다.

다음 편은 이 위에 올라가는 두 번째 시스템 — **7개 로케일을 어떻게 운영하는가** 얘기다.
