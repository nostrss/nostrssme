[이전 글](/blog/260609-pegboard-01-start)에서 Pegboard를 Next.js 16 정적 export로 짓는다고 했다. 이번 편은 그 결정이 처음 진짜로 아팠던 사건과, 그 후에 박은 게이트 얘기다.

## 정적 export 모드 한 줄 요약

`next.config.ts`에 한 줄 있으면 그게 끝이다.

```typescript
const nextConfig: NextConfig = {
  output: 'export',
  trailingSlash: true,
  images: { unoptimized: true },
  // ...
}
```

이 한 줄이 의미하는 건 *런타임에는 Next.js 서버가 없다.* `pnpm build`가 끝나면 `out/` 폴더에 정적 HTML/JS/CSS만 떨어지고, Cloudflare Pages는 그걸 그냥 서빙한다. API 라우트도 없고, 미들웨어도 없고, 서버 컴포넌트도 빌드 타임에 펼쳐져서 정적이 된다.

여기까지는 익숙한 얘기다. 문제는 다음 줄이다.

> **"클라이언트에서 도는 모든 코드는 브라우저-안전한 의존성만 import 해야 한다."**

이게 머리로는 당연한데 손은 늘 까먹는다.

## quicktype 사건

5월 30일에 JSON → TypeScript 타입 생성 페이지(`/json-to-typescript/`)를 추가했다. [`quicktype-core`](https://github.com/glideapps/quicktype) 라이브러리를 가져다 썼다. 잘 알려진 라이브러리고, JSON 샘플에서 깔끔한 TypeScript 타입을 만들어낸다.

- `pnpm typecheck` — 통과
- `pnpm lint` — 통과
- `pnpm test` (Vitest, Node 환경) — 통과
- `pnpm build` — 통과
- 배포 — 성공

그런데 사이트에 들어가서 `/json-to-typescript/` 를 누르면 **빈 화면**. 콘솔에는 이런 게 떠 있었다:

```
Uncaught Error: Cannot find module 'node:fs'
```

이유는 단순했다. `quicktype-core`가 모듈 최상단에서 `node:fs`를 import 한다. Node 환경에선 멀쩡하지만 브라우저에는 `node:fs`가 없다. 모듈 평가(module evaluation) 시점에 죽는다. **페이지가 마운트되기 전에 죽는다.**

| 검증 게이트          | 통과 여부 | 왜 못 잡았나                                |
| -------------------- | --------- | ------------------------------------------- |
| `typecheck`          | ✅        | 타입은 멀쩡, 환경 정보 없음                 |
| `lint`               | ✅        | 정적 분석은 import 환경을 따지지 않음       |
| `test` (Vitest/Node) | ✅        | **Node 환경**이라 `node:fs`가 있음          |
| `build`              | ✅        | 정적 export는 모듈 평가를 빌드 타임에 안 함 |
| 사이트 방문          | ❌        | 여기서 처음 죽는다                          |

진짜 게이트는 단 하나, **사람이 실제로 페이지를 열어보는 것**이었다. 이건 자동화가 아니다. 그래서 사고가 났다.

## 사후 처리 — quicktype 제거

사후 처리 커밋에서 `quicktype-core`를 완전히 빼고, 의존성 없는 순수 TypeScript로 JSON → TS 변환기를 다시 짰다. 같은 API를 유지하면서 브라우저 안전하게. diff 통계만 봐도 *순수 TS 재구현*의 무게가 보인다.

```
 package.json                          |   1 -
 pnpm-lock.yaml                        | 196 +-------------------------
 src/lib/json-to-typescript.ts         | 249 ++++++++++++++++++++++++++++++----
 tests/unit/json-to-typescript.test.ts |  46 +++++++
```

배열·선택적 필드·유니온 회귀 테스트를 추가했다. 하지만 이것만으로는 부족했다. **다음에 또 어떤 라이브러리가 같은 방식으로 죽을지 모른다.**

## 진짜 해결 — 레지스트리 구동 브라우저 스모크 게이트

페이지를 사람이 직접 열어보는 일을 사람이 안 해도 되게 만들어야 한다. 그래서 Playwright 스모크 테스트를 박았다 (`tests/e2e/utility-smoke.spec.ts`):

```typescript
import { utilities } from '../../src/utilities/registry';

const IGNORED_CONSOLE: readonly RegExp[] = [
  /Failed to load resource/i, // optional asset 404s — not a page-death signal
];

for (const { slug } of utilities) {
  test(`utility "${slug}" loads in the browser with no fatal errors`, async ({
    page,
  }) => {
    const errors: string[] = [];

    page.on('pageerror', (err) => {
      errors.push(`pageerror: ${err.message}`);
    });
    page.on('console', (msg) => {
      if (msg.type() !== 'error') return;
      const text = msg.text();
      if (IGNORED_CONSOLE.some((re) => re.test(text))) return;
      errors.push(`console.error: ${text}`);
    });

    const response = await page.goto(`/en/${slug}/`);
    expect(response?.ok()).toBe(true);
    await expect(page.locator('h1').first()).toBeAttached();
    expect(errors).toEqual([]);
  });
}
```

핵심은 세 가지다.

### 1. 레지스트리 기반 — 새 유틸이 자동으로 게이트에 들어간다

`for (const { slug } of utilities)` 한 줄로 끝난다. 유틸을 하나 추가하면 그 페이지의 브라우저 스모크 테스트가 *자동으로* 따라온다. 사람이 e2e 테스트 파일을 잊을 일이 없다. 레지스트리에 대해선 다음 편에서 자세히 다룬다.

### 2. 모듈 평가 사망을 잡는 두 신호

- `page.on('pageerror', ...)` — 모듈 평가 사망이 일으키는 uncaught exception
- `page.on('console', msg.type() === 'error', ...)` — 콘솔 에러

둘 중 하나라도 잡히면 fail. `quicktype-core` 사건 같은 *마운트 전에 죽는* 클래스의 버그가 정확히 여기 걸린다.

### 3. h1 attachment

```typescript
await expect(page.locator('h1').first()).toBeAttached();
```

`toBeVisible()`이 아니라 `toBeAttached()`다. `offline-ai-chat`처럼 `sr-only` H1을 쓰는 페이지가 있어서 시각적으로 보이지 않아도 DOM에 붙어있기만 하면 통과한다. 페이지가 마운트되었는지를 보는 신호다.

## 노이즈 필터 — 한 줄도 함부로 추가하지 말 것

`IGNORED_CONSOLE` 배열에는 단 한 줄만 있다.

```typescript
/Failed to load resource/i, // optional asset 404s — not a page-death signal
```

여기에 한 줄 추가할 때마다 "이건 *진짜로* 무해한가" 를 확인한다. 광고 스크립트도 가끔 콘솔에 noisy한 걸 찍지만 그건 별도 처리로 한다. 무해한 척하는 패턴을 ignore에 넣는 순간 게이트는 무력화된다.

> 진짜 어플리케이션 에러를 잠재우는 용도로 쓰지 말 것. (`utility-smoke.spec.ts:24-27` 주석)

## 이후의 룰 — AGENTS.md에 명문화

이 사건을 한 줄로 기록해뒀다 (`AGENTS.md`).

> **유틸 변환 로직은 브라우저-안전해야 한다.** 사이트는 정적 export 라서 모든 유틸의 변환·파싱 로직은 브라우저에서 돈다. Node-only 모듈(`node:fs` 등)을 모듈 스코프에서 import 하는 의존성은 *빌드는 통과하고 Node-env 유닛 테스트도 통과하지만* 브라우저에서 모듈-평가 에러로 죽는다 — `quicktype-core`가 정확히 이 방식으로 `/json-to-typescript/`를 조용히 깨뜨렸다.

이게 단순한 룰처럼 보이지만, 실은 **검증 전략의 우선순위**를 다시 잡으라는 신호다.

| 가짜 신호 (이게 통과해도 안심 X) | 진짜 게이트                          |
| -------------------------------- | ------------------------------------ |
| `typecheck`, `lint`              | `pnpm build` (정적 export 완료)      |
| `test` (Node 환경 Vitest)        | `pnpm test:e2e` (실제 브라우저 + 콘솔/page 에러 캡처) |
| 빌드 산출물 떨어짐                | h1 attached, 0 errors                 |

이 게이트는 **새 의존성 추가 후 가장 먼저 확인할 신호다.** 빌드가 통과해도 못 믿는다 — 브라우저가 통과해야 통과다.

## 비슷한 함정들 — 다음에 또 만날 수 있는 것들

자료를 좀 찾아보니 이 클래스의 함정이 꽤 흔하다.

- Node `Buffer`, `process.env`를 모듈 스코프에서 참조
- 서버 전용 라이브러리(`fs`, `path`, `crypto` Node 버전)를 직접 import
- 동적 import 안 쓰고 `require`로 가져오는 commonjs 라이브러리
- `polyfill`을 가정하고 짠 라이브러리 (예: 옛 `crypto-browserify` 의존 가정)

다행히 위 모든 케이스가 *브라우저 스모크 테스트*에 걸린다. 페이지를 열어보고 콘솔에 빨간 줄이 뜨면 거기서 끝난다. 이게 자동화되어 있다는 게 핵심이다.

## 정리

- Next.js 정적 export(`output: 'export'`)의 가장 위험한 함정은 **빌드도 통과하고 유닛 테스트도 통과하는데 브라우저에서 죽는 의존성**이다.
- `quicktype-core`가 모듈 최상단에서 `node:fs`를 import 해서 `/json-to-typescript/`가 빈 페이지가 됐다. 5단계 검증 게이트가 다 통과했는데도 사이트가 깨졌다.
- 진짜 게이트는 **레지스트리 구동 Playwright 스모크 테스트** — `pageerror`와 `console.error`를 캡처해 새 유틸이 추가되어도 자동으로 게이트에 들어간다.
- `IGNORED_CONSOLE` 노이즈 필터에는 *진짜로 무해한* 패턴만 넣는다. 한 줄 늘 때마다 게이트가 약해진다.
- 룰: 새 의존성 추가 시 `pnpm build` 통과를 신뢰하지 마라. **브라우저가 통과해야 통과다.**

다음 편은 이 게이트를 자동으로 굴리는 핵심 부품 — **유틸리티 레지스트리** — 얘기다.
